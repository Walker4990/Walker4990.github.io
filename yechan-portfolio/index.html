👟 TROUBLE SHOOTING

🚨 문제 배경

출고 등록(newOutInven)은 정상 동작했지만, 출고 수정(updateOutInven) 요청 시 JMeter 부하 테스트에서 500 에러 100% 발생.

  • 평균 응답시간 297ms 이상 → 서비스 지연 심각
  • NullPointerException 발생 (stockQuantity 값 누락)
  • DB에서 풀스캔 발생 → 불필요한 부하
  • 동시성 제어 부재 → 재고 음수 발생 위험

📌 개선 방향

  • DB 인덱스 최적화 및 쿼리 조건 강화 → 풀스캔 제거
  • 출고 수정 로직 개선 → NPE 방지 및 원자적 재고 갱신 보장
  • 재무/세금/로열티는 반드시 정정 INSERT → 회계 투명성 확보
  • JMeter 부하 테스트로 사용자 수별 성능 곡선 측정

⭐ 해결 방법

1. 복합 인덱스 생성


-- Before (풀스캔 발생)
SELECT * FROM shipment WHERE inventory_no = ?;

-- After (복합 인덱스 적용)
CREATE INDEX idx_shipment_inven_sn 
ON shipment (inventory_no, shipment_no DESC);

2. 출고 수정 쿼리 보완 (단일 행 보장)


-- Before (다건 수정 위험)
WHERE inventory_no = #{inventoryNo}

-- After (단건 수정 보장)
WHERE shipment_no = #{shipmentNo}
AND inventory_no = #{inventoryNo}

3. 재고 차감 원자화 처리 (음수 방지)


UPDATE inventory
SET quantity = quantity - #{editQuantity}
WHERE inventory_no = #{inventoryNo}
AND quantity - #{editQuantity} >= 0;

4. 서비스 계층 개선 (@Transactional)


// Before
Integer authorNo   = mapper.getAuthorNo(bookNo);
Integer contractNo = mapper.getContractNo(bookNo);
BigDecimal royalty = mapper.getRoyaltyRate(bookNo);

// After
Map<String, Object> meta = mapper.getBookMeta(bookNo);

🤩 알게 된 점

  • DB와 애플리케이션의 병목 원인 구분 능력이 중요
  • 성능 개선은 단순한 속도 향상이 아니라 정합성 + 안정성 + 확장성까지 함께 보장해야 함

🚨 문제 배경

  • 대시보드 화면에서 사용하는 /getInventoryBook API는 책별 재고 수량을 집계해서 보여주는 기능이다.
  • 처음에는 단순히 DB에서 직접 조회하도록 구현했는데, 사용자가 늘어나면 같은 API가 반복 호출되면서 DB에 부하 발생.
  • JMeter 부하 테스트 결과:
    • 동시 사용자 100명 → 평균 응답속도 40ms 이상
    • 500명 이상 → 응답속도 증가 및 Throughput 한계 발생
  • 사실 데이터는 자주 바뀌지 않는데도 매번 DB에서 가져와 비효율적 문제가 있었다.

📌 개선 방향

  • /getInventoryBook API는 단순 재고 현황 조회 기능 → 매 호출마다 DB 접근은 비효율적
  • 자주 조회되지만 큰 변화가 없는 데이터 → Redis 캐시 활용
  • Spring @Cacheable 적용, TTL(10분) 설정으로 데이터 신선도 보장
  • JMeter로 캐시 전/후 성능 비교 측정 → 개선 효과 수치화

⭐ 해결 방법

3. 급여 등록 / 지급 UX 개선

🚨 문제 배경

직원 수가 많아질수록 급여 등록을 개별 입력으로 처리하는 것은 비효율적이었고, 보너스/OT 입력 시 값이 테이블에 실시간 반영되지 않아 재입력·새로고침이 반복되는 문제가 발생.

⭐ 해결 방법

급여 담당자 UX 개선 목표: “등록 → 지급 → 확인” 과정을 한눈에 직관적으로 볼 수 있도록 설계.

  • 급여 등록
    • 실시간 검색 자동 추가: jQuery Ajax 기반으로 이름/사번 입력 시 Ajax 호출 → 검색 결과를 하단 테이블에 append()
    • 보너스 입력 시 실시간 계산: .on("input") 이벤트로 세금·총액 즉시 재계산 → 테이블과 hidden input 값에 반영
    • 세금 계산 자동화:
      • 과세 소득 = 기본급 + 직책 수당 + 보너스 + OT
      • 공제 = 국민연금(4.5%) + 건강보험(3.5%) + 고용보험(0.9%) + 소득세(8%) + 지방세(10%)
      • 실수령액 = 과세 소득 + 식대 – 공제
    • 데이터 보존: 입력 값을 bonusMap에 저장해 재검색 시에도 값 유지
  • 급여 지급
    • 일괄 지급 처리: 전 직원 데이터를 Ajax + JSON(FormData)로 전송 → Spring Controller @RequestBody로 리스트 수신
    • 다건 Insert 처리: MyBatis <foreach> 구문으로 payroll 테이블에 다건 insert
    • 지급 상태 확인: 지급 완료 후 JSP 테이블의 .statusField 업데이트 → 관리자 즉시 확인

✅ 성과

  • 급여 등록: 실시간 검색 → 선택 → 입력 → 계산이 한 화면에서 가능
  • 급여 지급: 일괄 처리 + 지급 상태 확인으로 실제 인사·회계 팀의 업무 흐름과 일치

🤩 알게 된 점

  • 실시간 반영 로직이 UX 개선에 미치는 효과 체감
  • 다건 insert + 트랜잭션 적용으로 안정적인 급여 지급 보장
  • 업무 프로세스를 단순 기능 구현이 아닌 사용자 관점(UX)에서 설계하는 것의 중요성 학습

4. 품질 검수 / 클레임 처리 로직 혼선

🚨 문제 배경

초기에는 발주 완료 즉시 재고에 등록되도록 구현했는데, 이렇게 하면 검수 과정이 생략되어 불량 수량이 누락되고 재무/재고 데이터가 왜곡되는 문제가 발생.

📌 개선 방향

  • 현업에서는 배송사 API 연동으로 배송완료 자동 처리가 가능하지만, 프로젝트 환경에서는 실제 판매·배송 데이터가 없어 API 사용 불가
  • 따라서 개발 단계에서는 배송완료 상태를 직접 입력하도록 설계 → 이후 검수 테이블로 이동 보장
  • 흐름: 발주 → 배송 → 배송완료 시 검수 → 검수 완료 후 수량 입력 → 합격 수량만 재고 등록 → 부족 수량은 추가 발주

⭐ 해결 방법

  • 배송 완료 버튼을 눌러야 검수 가능하도록 조건 분기
  • 검수 시: 합격 수량만 재고 IN, 불량 수량은 품질 데이터에만 기록 (재무 반영 ❌)
  • 클레임 처리 시:
    • 회수 가능 + 회수 완료 → 재고 IN + 환불 처리
    • 회수 불가 → 환불 처리만, 재고에는 반영 ❌
  • Service 계층에서 분기 로직 구현, Mapper XML에서 상태값별 SQL 분리

🤩 알게 된 점

  • 비즈니스 로직은 DB보다 서비스 계층에서 관리하는 것이 유지보수에 유리
  • 트랜잭션이 데이터 정합성 보장의 핵심임을 직접 경험
  • 실패 시나리오까지 검증하는 습관이 안정적 시스템 운영에 필수
  • SQLD 학습 개념이 실제 프로젝트 의사결정에 실질적으로 도움이 됨
